Sobre el scheduler de automatizacion

En produccion app.py intenta from src.automation import start_scheduler; start_scheduler(), pero esa funcion no existe hoy en automation.py (solo hay un BackgroundScheduler con el add_job comentado). El ImportError se captura y se imprime "Could not start scheduler". Resultado real: no corren tareas en background. Todas las campanas masivas se disparan manualmente desde el panel admin. Si en el futuro se quiere automatizar, hay que implementar start_scheduler() en automation.py y registrar el job.

7 flows.py — Maquina de Estados (FlowHandler)

Este es el archivo mas critico del bot. Contiene toda la logica de procesamiento de mensajes.

Clase: FlowHandler (todos metodos @staticmethod)

handle_incoming_message(payload)

Entry point. Extrae datos del payload de Meta, identifica tipo de mensaje, descarga media si aplica, y rutea a process_text_input() o process_button_click().

Procesamiento de media (imagen/documento):

  1. Extrae media_id del payload
  2. Llama WhatsAppService.get_media_url(media_id) para obtener URL temporal de Meta
  3. Descarga a static/uploads/{phone}/{filename}
  4. Sube a Supabase Storage bucket "media" en path {phone}/{filename}
  5. Registra en bot_messages con la URL publica de Supabase
  6. Si estado es waiting_for_docs_rojo o waiting_for_cuenta_amarillo: registra en received_documents y programa confirmacion con debounce de 60s

Camino de envio: _send_to_meta()

Todos los metodos send_* terminan en este helper central. Antes de salir a la red atraviesan tres compuertas en orden:

1. test_mode.is_test_phone(to)  → si es telefono __test_, NO sale a Meta:
                                  acumula el outbound en el buffer en memoria.
2. _staging_blocks_send(...)    → si ENVIRONMENT=staging, NO sale a Meta:
                                  imprime "[STAGING] Envio real bloqueado" y retorna.
3. (image/document) verify_remote_media(url, kind) → valida el link ANTES de enviar.

_send_to_meta(to, payload, kind, log_text, log_type):
  - POST con timeout 15s y reintentos: _SEND_ATTEMPTS=3 (1 + 2 retries),
    backoff (1.5s, 3.0s). Solo reintenta errores TRANSITORIOS (5xx, red,
    rate limit); rechazos definitivos (token, ventana 24h, payload) NO.
  - Exito  → log_message(...) con wamid, devuelve JSON de Meta.
  - Fracaso→ error_tracker.record_event(categoria) + log_message(..., "failed")
             (visible en el panel) y devuelve None. Clasificacion en error_tracker.

Validacion de media — evita el error Meta 131053

Por que existe esto

Al enviar imagenes/documentos por link, Meta valida el archivo de forma asincrona: acepta el envio (HTTP 200 + wamid) y recien despues intenta descargar la URL. Si el archivo esta vacio (0 bytes), excede el limite de tamano o no es accesible, Meta lo rechaza con el codigo 131053 ("No media found from weblink and received http code 200") via webhook de estado — y el asesor ya creyo que el documento salio. Un caso real (jun/2026): un PDF subido por el panel quedo en Supabase con Content-Length: 0 y reboto.

META_MEDIA_LIMITS = {"image": 5 MB, "document": 100 MB}

verify_remote_media(url, kind) -> (ok: bool, motivo: str | None)
  # HEAD a la URL (timeout 15s, sigue redirects). Bloquea el envio si:
  #   - no responde 200
  #   - Content-Length == 0  (archivo vacio)
  #   - Content-Length > limite de Meta para ese tipo
  # Sin Content-Length, deja pasar (no bloquea envios legitimos).

# send_image / send_document llaman verify_remote_media tras el guard de
# staging. Si falla: error_tracker.record_event("codigo", ...) +
# log_message(..., "failed") y retornan None (NO se envia a Meta).

Defensa en capas complementaria en la subida (admin.py api_upload_media rechaza 0 bytes y archivos sobre el limite con HTTP 400) y en upload_to_supabase_storage (no sube bytes vacios). Asi el problema se corta en el origen y, si algo se cuela, antes de llegar a Meta.

Log de mensajes threading.Thread(daemon=True) conversation_log.py Cada log_message() lanza un thread para no bloquear el webhook response
Confirmacion de docs threading.Timer(60, fn) automation.py Debounce: cancela timer anterior, espera 60s desde ultimo doc
Enriquecimiento empresa threading.Thread(daemon=True) conversation_log.py Busca empresa via Cloud Run para docs sin empresa registrada
Consideracion importante

Los threads daemon=True se matan si el proceso principal termina. En Render con Gunicorn, los workers pueden reiniciarse. Los timers de debounce se pierden en un restart. Esto es aceptable porque el peor caso es que no se envie la confirmacion de documentos (no se pierde data).

24 Despliegue

Bot principal (Render)

  1. Push a main en el repo Git conectado a Render
  2. Render detecta el cambio y hace build automatico
  3. Build: instala requirements.txt
  4. Start: ejecuta Procfile → gunicorn app:app --bind 0.0.0.0:$PORT
  5. Variables de entorno configuradas en Render Dashboard → Environment

Cloud Run (Microservicio DB)

  1. Archivo: cloud_run/main.py
  2. Deploy via gcloud functions deploy o Cloud Run Deploy
  3. Variables: DB_HOST, DB_NAME, DB_USER, DB_PASSWORD, DB_PORT, API_TOKEN_SECRET
  4. Framework: functions-framework
  5. Entry point: get_solicitud

Configuracion de Webhook en Meta

  1. Meta Developer Portal → App → Webhooks
  2. Callback URL: https://[tu-dominio-render]/webhook
  3. Verify token: valor de WEBHOOK_VERIFY_TOKEN
  4. Suscribir a: messages
  5. El webhook GET verifica el token, POST recibe mensajes

27 Como Extender el Bot (Guia para Desarrolladores)

Agregar un nuevo estado al flujo

  1. Definir el nombre del estado (ej: waiting_for_referencia)
  2. En flows.py → process_text_input(): agregar un nuevo elif state == "waiting_for_referencia":
  3. Implementar la logica: validar input, guardar datos, transicionar al siguiente estado
  4. Agregar la transicion de entrada (donde se hace set_user_state(phone, "waiting_for_referencia"))

Agregar un nuevo boton interactivo

  1. En el metodo que envia el menu, agregar el boton: {"id": "mi_nuevo_btn", "title": "Mi Opcion"} (max 3 botones por mensaje, titulo max 20 chars)
  2. En flows.py → process_button_click(): agregar elif btn_id == "mi_nuevo_btn":
  3. Implementar la accion del boton

Agregar una nueva tabla en Supabase

  1. Crear la tabla en Supabase Dashboard → Table Editor
  2. Configurar RLS (Row Level Security) si es necesario
  3. En conversation_log.py: agregar funciones CRUD usando supabase_client.table("mi_tabla")
  4. Importar y usar desde flows.py o admin.py

Agregar un nuevo template / campana de WhatsApp

  1. Crear el template en Meta Business Manager → Message Templates y esperar aprobacion (hasta 24h)
  2. En automation.py: crear get_pending_*() y execute_bulk_*() siguiendo el patron existente (limpieza de telefono, filtros de ya-notificado-hoy, send_template, set de estado destino)
  3. En admin.py: agregar rutas GET (elegibles), POST (envio) y, si aplica, *-metrics
  4. En static/js/campaigns.js: agregar la config declarativa de la campana (la UI la genera la factory campaign-panel.js o paste-campaign.js, no se escribe HTML a mano)
  5. En test_mode.TEMPLATES: agregar el cuerpo y botones para poder probarla en /admin/test
  6. Probar en staging antes de mergear a main

Agregar una nueva ruta en el panel admin

  1. En admin.py: agregar la ruta con @admin_bp.route() y @requires_auth
  2. En admin.html: agregar el fetch() correspondiente y la UI

Modificar el comportamiento del LLM

  1. Sin tocar codigo: Editar los archivos en context/. Especialmente capacidades_bot.md y ejemplos_conversacion.md
  2. Cambiando la logica: Editar _SYSTEM_PROMPT en llm.py (lineas ~43-139)
  3. Cambiando el modelo: Cambiar "claude-haiku-4-5-20251001" por otro modelo en ask_llm()
  4. Agregar un nuevo tag: Definir en el system prompt, detectar en flows.py dentro del handler de agent_llm

Agregar una nueva consulta al Cloud Run

  1. En cloud_run/main.py: agregar un nuevo elif tipo == "mi_consulta":
  2. Escribir la query SQL contra las vistas disponibles
  3. Redesplegar Cloud Run: gcloud functions deploy get_solicitud --runtime python311 --trigger-http
  4. En database.py: agregar funcion get_mi_consulta()

Patrones clave a seguir

  • Probar en staging primero: todo cambio que toque src/ pasa por la rama staging y el panel /admin/test antes de main. Cambios solo-panel (HTML/CSS/JS) pueden ir directo.
  • No tocar flows.py sin confirmacion: afecta el comportamiento en produccion.
  • El bot nunca admite ser IA: ningun cambio puede revelarlo.
  • Logging asincrono: Siempre usar log_message() — nunca bloquear el webhook por logging
  • Retornos None: Las funciones de database.py retornan None en error. Siempre validar antes de usar
  • Media por link: validar tamano/accesibilidad antes de enviar (verify_remote_media) — Meta falla async (131053) y deja falsos "enviado".
  • Soporte test_mode: al agregar un envio o una escritura a Supabase, respetar el corto-circuito de test_mode para que el simulador siga funcionando.
  • Estado como string: Los estados se guardan como string en Supabase. Usar nombres descriptivos con snake_case
  • Telefonos con 57: Normalizar siempre al formato 57XXXXXXXXXX
  • Templates: Los nombres deben coincidir exactamente con los aprobados en Meta; reflejar el cuerpo en test_mode.TEMPLATES
  • Errores visibles: ante un fallo de envio/integracion, registrar con error_tracker.record_event() para que aparezca en Diagnostico
  • Idioma: Todo en espanol colombiano, tuteo, tono profesional pero cercano, sin emojis ni negrillas en respuestas del bot
